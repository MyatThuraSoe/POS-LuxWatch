# Phase 1: Authentication & Role Management

## Overview
**Phase Objective:** Implement secure authentication, token lifecycle management, and a granular Role-Based Access Control (RBAC) system separating ADMIN, OWNER, and EMPLOYEE roles.
**Problem Solved:** Prevents unauthorized access, enforces principle of least privilege, and establishes a scalable permission model that keeps OWNER and ADMIN logically separate despite initial overlap.
**User Interaction:** Users log in via React frontend, receive Sanctum tokens, and access role-gated routes. Session management handles token refresh and logout.
**Backend Architecture Considerations:** Laravel Sanctum for token issuance. Custom `Role` and `Permission` models with many-to-many pivots. Middleware: `auth:sanctum`, `role:admin,owner,employee`, `permission:*`. Policy classes for model-level authorization. Token scopes for device-specific access.

## Business Requirements
- Secure login/logout with token-based authentication.
- Distinct roles: ADMIN (system config, user management), OWNER (financial oversight, full business control), EMPLOYEE (POS operations, limited data access).
- Granular permissions assignable to roles.
- Token expiration and revocation capabilities.

## Functional Requirements
- Login endpoint returning Sanctum token + user profile + roles/permissions.
- Logout endpoint revoking current token.
- Token refresh mechanism.
- CRUD for roles and permissions (Admin/Owner only).
- Role assignment to users.
- Permission caching for fast middleware checks.
- Failed login tracking and account lockout.

## User Stories
- As an Employee, I want to log in with my credentials so I can access the POS terminal.
- As an Owner, I want to assign specific permissions to roles so I can control what staff can do.
- As an Admin, I want to revoke compromised tokens so I can secure the system.

## Use Cases
- **User Login:** Submit credentials → Validate → Issue Sanctum token → Return user context + permissions → Cache permissions.
- **Role Assignment:** Admin selects user → Assigns role → Updates pivot table → Clears permission cache.
- **Token Revocation:** User logs out or Admin revokes → Token deleted from `personal_access_tokens` → Subsequent requests return 401.

## Permissions Matrix
| Role     | auth.login | auth.logout | roles.manage | permissions.manage | tokens.revoke |
|----------|------------|-------------|--------------|--------------------|---------------|
| ADMIN    | ✓          | ✓           | ✓            | ✓                  | ✓             |
| OWNER    | ✓          | ✓           | ✓            | ✓                  | ✓             |
| EMPLOYEE | ✓          | ✓           | ✗            | ✗                  | ✗             |

## Database Design
### Tables & Columns
- `users`: (extends in Phase 2)
- `roles`: `id`, `name` (unique), `guard_name`, `description`, `created_at`, `updated_at`
- `permissions`: `id`, `name` (unique), `guard_name`, `module`, `description`, `created_at`, `updated_at`
- `role_user`: `role_id`, `user_id`, `assigned_by`, `created_at`
- `permission_role`: `permission_id`, `role_id`, `created_at`
- `personal_access_tokens`: (Sanctum default)

### Relationships
- `User` ↔ `Role` (Many-to-Many via `role_user`)
- `Role` ↔ `Permission` (Many-to-Many via `permission_role`)
- `User` → `PersonalAccessToken` (One-to-Many)

## Validation Rules
- Email: `required|email|max:255|exists:users,email`
- Password: `required|string|min:8|regex:/^(?=.*[a-z])(?=.*[A-Z])(?=.*\d)(?=.*[@$!%*?&])[A-Za-z\d@$!%*?&]/`
- Role/Permission names: `required|string|max:50|alpha_dash|unique`
- Token name: `required|string|max:100`

## Business Rules
- OWNER and ADMIN roles are hardcoded seeds but remain editable for permissions.
- EMPLOYEE cannot modify roles or permissions.
- Tokens expire after 12 hours of inactivity.
- 5 failed login attempts within 15 minutes locks account for 30 minutes.
- Permissions are cached for 60 minutes; cache busts on role/permission changes.

## API Specification
### Endpoints
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/logout`
- `GET /api/v1/auth/me`
- `POST /api/v1/auth/refresh`
- `GET /api/v1/roles`
- `POST /api/v1/roles`
- `GET /api/v1/permissions`
- `POST /api/v1/users/{id}/roles`

### Request Examples
```json
POST /api/v1/auth/login
{
  "email": "cashier@watchshop.com",
  "password": "SecurePass123!",
  "device_name": "POS-Terminal-01"
}
```

### Response Examples
```json
{
  "success": true,
  "data": {
    "token": "3|AbCdEfGhIjKlMnOpQrStUvWxYz",
    "user": { "id": 5, "name": "John Doe", "email": "cashier@watchshop.com" },
    "roles": ["EMPLOYEE"],
    "permissions": ["pos.sales.create", "pos.cart.manage"]
  },
  "message": "Authenticated successfully",
  "errors": null,
  "meta": { "expires_in": 43200 }
}
```

### Error Cases
- `401 Unauthorized`: Invalid credentials or expired token
- `403 Forbidden`: Insufficient permissions for role management
- `423 Locked`: Account temporarily locked due to failed attempts
- `422 Unprocessable Entity`: Validation failure

## Security Requirements
- Passwords hashed with `bcrypt` (cost 12) or `argon2id`.
- Sanctum tokens stored hashed in DB.
- Brute force protection via login throttling.
- Role/Permission middleware checks on every protected route.
- Audit all authentication events and role changes.
- Prevent privilege escalation via mass assignment protection.

## Acceptance Criteria
- Users can authenticate and receive valid Sanctum tokens.
- RBAC middleware correctly restricts routes based on roles/permissions.
- OWNER and ADMIN roles are distinct and independently configurable.
- Token refresh and revocation work as specified.
- Failed login lockout mechanism triggers correctly.
- Permission caching improves middleware response time.

## Future Enhancements
- Multi-factor authentication (TOTP/SMS).
- Device fingerprinting for trusted terminals.
- SSO/OIDC integration for corporate accounts.
- Session concurrency limits per user.

# Phase 4: Inventory Management

## Overview
**Phase Objective:** Implement real-time stock tracking, serial number management, stock adjustments, and low-stock alerting for watch variants.
**Problem Solved:** Prevents overselling, ensures accurate financial valuation, tracks individual high-value watches via serial numbers, and automates replenishment triggers.
**User Interaction:** Staff view stock levels, perform stock takes, adjust quantities, and receive low-stock warnings. System automatically deducts stock on sale completion.
**Backend Architecture Considerations:** PostgreSQL transactions for atomic stock updates. Row-level locking (`lockForUpdate()`) during checkout. Event-driven architecture: `StockDepleted`, `LowStockAlert`. Repository pattern for complex inventory queries. Redis caching for fast POS stock checks.

## Business Requirements
- Track quantity per variant.
- Manage serial numbers for individual watches (critical for warranty).
- Record all stock movements (sales, purchases, adjustments, returns).
- Support manual stock adjustments with reason codes.
- Trigger alerts when stock falls below threshold.
- Prevent negative stock without explicit override.

## Functional Requirements
- Real-time stock level API.
- Serial number registration and status tracking (In Stock, Sold, Returned, Defective).
- Stock adjustment workflow with approval (optional).
- Low stock threshold configuration per product/variant.
- Stock movement history with audit trail.
- CSV export for physical stock takes.

## User Stories
- As an Employee, I want to see available stock before adding to cart so I don't promise unavailable items.
- As an Owner, I want to track serial numbers so I can honor warranties accurately.
- As an Admin, I want to adjust stock after a physical count so the system matches reality.

# Phase 9: Customer & Warranty Management

## Overview
**Phase Objective:** Manage customer profiles, link sales to customers, track warranty periods, and handle warranty claims/repairs for after-sales service.
**Problem Solved:** Enables loyalty building, streamlines warranty validation, tracks repair jobs, and ensures compliance with manufacturer/service terms.
**User Interaction:** Cashier links sale to customer. Service staff register warranty claims, track repair status, and notify customers upon completion.
**Backend Architecture Considerations:** Customer PII stored securely. Warranty engine calculates expiry based on sale date + product terms. Repair workflow state machine. Event-driven notifications (email/SMS stubs). Repository pattern for claim filtering and SLA tracking.

## Business Requirements
- Create and manage customer profiles.
- Link sales to customers for purchase history.
- Auto-generate warranties based on product terms.
- Track warranty status (Active, Expired, Void, Claimed).
- Manage repair jobs with status tracking and cost logging.
- Validate warranty claims against purchase date and serial.

## Functional Requirements
- Customer CRUD with search and purchase history.
- Warranty registration and expiry calculation.
- Claim submission with validation rules.
- Repair job tracking (Received, Diagnosing, Repairing, Ready, Completed, Cancelled).
- Customer communication logging.
- Warranty voiding rules (tampering, unauthorized repair).

## User Stories
- As a Cashier, I want to attach a customer to a sale so they can receive warranty coverage.
- As a Service Tech, I want to validate a warranty claim so I can approve free repairs.
- As an Owner, I want to track repair turnaround times so I can improve service SLAs.

## Use Cases
- **Warranty Registration:** Sale completes → System checks product warranty term → Creates warranty record linked to serial & customer → Sets expiry date.
- **Claim Validation:** Customer brings watch → Staff scans serial → System checks warranty status & expiry → Approves/Rejects claim → Creates repair job.
- **Repair Completion:** Tech finishes repair → Updates status to Ready → System notifies customer → Customer picks up → Status set to Completed.

## Permissions Matrix
| Role     | customers.view | customers.create | warranties.view | warranties.claim | repairs.manage |
|----------|----------------|------------------|-----------------|------------------|----------------|
| ADMIN    | ✓              | ✓                | ✓               | ✓                | ✓              |
| OWNER    | ✓              | ✓                | ✓               | ✓                | ✓              |
| EMPLOYEE | ✓              | ✓                | ✓               | ✓                | View only      |

## Database Design
### Tables & Columns
- `customers`: `id`, `name`, `email`, `phone`, `address`, `notes`, `created_at`, `updated_at`
- `warranties`: `id`, `sale_id`, `serial_id`, `customer_id`, `start_date`, `end_date`, `status` (enum: active, expired, void, claimed), `terms`, `created_at`, `updated_at`
- `warranty_claims`: `id`, `warranty_id`, `claim_number`, `reason`, `status` (enum: pending, approved, rejected), `submitted_by`, `submitted_at`, `resolved_at`
- `repair_jobs`: `id`, `claim_id`, `serial_id`, `status` (enum: received, diagnosing, repairing, ready, completed, cancelled), `diagnosis`, `repair_cost`, `technician_id`, `created_at`, `updated_at`

### Relationships
- `Customer` → `Sale` (One-to-Many)
- `Customer` → `Warranty` (One-to-Many)
- `Warranty` → `WarrantyClaim` (One-to-Many)
- `WarrantyClaim` → `RepairJob` (One-to-One)
- `SerialNumber` → `Warranty` (One-to-One)

## Validation Rules
- Customer Email/Phone: Unique, valid format
- Warranty Dates: `end_date` > `start_date`
- Claim Reason: `required|string|max:500`
- Repair Cost: `numeric|min:0`
- Status Transitions: Enforced via workflow validation

## Business Rules
- Warranty auto-generated on sale if product has warranty term > 0.
- Claims rejected if warranty expired, voided, or serial mismatch.
- Void warranty if tamper seal broken or unauthorized service detected.
- Repair costs billed to customer if warranty expired/rejected.
- SLA tracking: alerts if repair exceeds configured days (e.g., 7 days).
- Customer PII access logged for compliance.

## Acceptance Criteria

- Warranty expiry calculates accurately from sale date.
- Claims validate against warranty status and serial correctly.
- Repair workflow transitions enforce business rules.

- All PII access and modifications are audited.

# Phase 11: Production Hardening & System Administration

## Business Requirements
- Manage system-wide settings (store info, tax defaults, currency, operating hours).
- Maintain immutable audit logs for all critical actions.
- Automate database and file backups.
- Monitor queue, cache, and job health.
- Provide administrative controls for cache clearing and maintenance mode.
- Ensure compliance and data retention policies.

## Functional Requirements
- System settings CRUD with validation.
- Audit log viewer with advanced filtering.
- Backup trigger and status tracking.
- Queue/Cache stats endpoint.
- Maintenance mode toggle.
- Log rotation and archival configuration.

## User Stories
- As an Owner, I want to update store hours and tax rates so the POS reflects current business rules.
